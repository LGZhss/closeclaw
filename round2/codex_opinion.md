## Codex 意见

### 议题1：动态法定人数
- 我的观点：支持分级方案，但需要“可解释的影响评分+硬阈值”的混合策略。建议低/中/高影响对应 N=3/5/7，并允许在极低风险与紧急场景下临时降至 N=2（需审计记录）。
- 理由：N=3/5/7 与 Round 1 共识相符，便于落地；再叠加影响评分（复杂度、风险、可逆性）可避免僵化。
- 建议方案：
  - 影响评分 S ∈ [1,10]，由任务复杂度、影响面、可逆性、合规风险四维加权得到。
  - N 规则：S≤3 → N=3；4-6 → N=5；≥7 → N=7。
  - 初始引导期：前 2 周或前 30 个决策采用“低一档”策略（N-2，最低 N=3），并设置强制复盘清单；达到稳定指标后自动退出引导期。
  - 自动切换监控指标：任务失败率/回滚率、专家一致性分数、置信度均值、S 分布漂移、单点主导率（HHI）。
  - 熔断触发：7 日滚动回滚率 > 8% 或高影响任务失败率 > 3%，或一致性分数连续 5 次低于阈值，触发升级到 N=7 并启用人工审阅。

### 议题2：动态权重调整
- 我的观点：支持混合权重模型（基础权重 + 绩效权重），而非纯绩效权重。
- 理由：纯绩效易导致短期波动放大与权重集中；混合模型更稳健，且更符合 Round 1 对“动态调节”的共识。
- 建议方案：
  - 基础:绩效 = 70%:30%，α=0.25（介于 20-30% 中位），采用逐步更新避免震荡。
  - 绩效维度：质量（正确性/完整性/可维护性）、速度（响应时延）、可靠性（失败率/回滚率）、创新性（差异化贡献）、协作度（一致性与复用率）。
  - 历史评分收集：采用 30-60 天滑动窗口，衰减因子 0.9-0.95；缺失数据用贝叶斯先验补齐。
  - 归一化：先对各维做 robust z-score，再做 min-max 到 [0,1]；最终绩效分做 softmax 温度缩放，避免极端值。
  - 防止集中：设置权重上限 0.25、下限 0.05；若 HHI>0.22 或 Top1 连续 10 次占比>35%，触发分散化惩罚（entropy regularization）。

### 议题3：沙盒隔离
- 我的观点：倾向“进程级隔离优先，隔离强度逐步加码”。在 Windows 环境，isolated-vm 虽可用但依赖原生模块与工具链，落地成本偏高，应先走 Worker Threads/子进程 + 资源限额，再评估 isolated-vm。
- 理由：vm2 维护与安全风险高；Worker Threads+策略隔离更易落地；高危执行需 OS 级隔离以降低逃逸风险。
- 建议方案：
  - 方案优先级：
    1) 子进程 + IPC + 资源限制（超时/内存/CPU）作为基线隔离。
    2) Worker Threads + `vm` 仅用于可信脚本与受控 API。
    3) isolated-vm 作为增强隔离的二期方案（先在 CI/Windows 构建环境做 POC）。
  - 降级方案：isolated-vm 构建失败或性能不达标时，回退到子进程模式并限制可用 API。
  - 实施阶段：
    - Phase 1（2-3 周）：子进程沙盒 + 超时/内存限额 + 审计日志。
    - Phase 2（2 周）：isolated-vm POC，验证 Windows 兼容与性能。
    - Phase 3（持续）：高危任务切换到更强隔离（可外置微服务/容器化）。
  - 风险预案：原生模块构建失败、性能回退、逃逸风险；对应预案为自动切换/禁用高危功能/强制人工审批。

### 其他建议
- 建议在 Round 2 结束前明确“影响评分 S 的定义与权重”，否则动态法定人数会变成主观判断。
- 权重与法定人数的调整都应记录到可追溯审计日志，便于 Round 3 验证。

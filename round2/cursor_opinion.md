# cursor 意见

### 议题1：动态法定人数
- **我的观点**：支持“分级（3/5/7）为主 + 公式化评分为辅”的混合方案；将“关键任务”单独定义为极少数场景才需要的 N=9（可选且默认不启用）。
- **理由**：
  - **可解释**：3/5/7 的分级对参与者最直观，便于协作与审计；纯公式方案在早期容易因阈值争议产生摩擦。
  - **可自动化**：在分级基础上引入“影响评分”用于自动推荐等级与复核触发，而不是直接决定。
  - **防极端**：N=9 若泛化会拖慢迭代；应仅用于“治理/安全/沙盒/权重引擎”等高风险变更。
- **建议方案**：
  - **分级方案（默认）**：
    - **低影响**：N≥3（文档、注释、小修复、非核心配置、低风险依赖升级）
    - **中影响**：N≥5（功能迭代、重构、适配器接口调整、缓存/内存策略调整）
    - **高影响**：N≥7（核心调度/投票/仲裁/安全/沙盒/权限/数据脱敏相关变更）
    - **关键影响（可选）**：N≥9（“决策权重计算引擎/投票仲裁器”规则变更、沙盒逃逸类修复、密钥/权限模型变更）
  - **初始引导期设置**（建议“时间 + 活跃主体数”双条件，避免只按时间硬切）：
    - **引导期（前14天或活跃主体 < 7）**：
      - 低影响 N≥2；中影响 N≥3；高影响 N≥5；关键影响仍 N≥7（不建议放到 2/3）
      - 每个决策必须附带“影响等级+信心度(1-5)”元数据
    - **稳定期（活跃主体 ≥ 7 且连续7天投票参与率≥70%）**：切回默认分级（3/5/7，关键可选9）
  - **自动切换监控指标**（用于“推荐等级/触发复核/触发熔断”，不建议完全无人值守强切）：
    - **变更规模**：改动文件数、净变更行数、是否涉及二进制/构建链
    - **影响面**：受影响模块数、是否触达 `core/`、`security/`、`sandbox/`、`weightCalculator/`、`arbitrator/`
    - **风险信号**：测试是否通过、静态检查结果、历史同类变更故障率、是否包含权限/密钥/执行入口变更
    - **协作信号**：可用主体数、过去24h响应延迟 P95、投票分歧度（如赞成/反对接近且理由冲突）
  - **熔断机制触发条件**（建议保留“三态熔断器”，并补齐触发与恢复条件）：
    - **触发 OPEN**：
      - 在投票窗口内未达到法定人数（例如 30-60 分钟，按影响级别配置）
      - 连续 3 次出现“未达法定人数”或“强分歧 + 无法仲裁”的失败
      - 监控显示系统健康恶化（例如响应超时/错误率暴涨）导致投票质量不可控
    - **HALF_OPEN 恢复**：
      - 活跃主体数恢复到阈值，且连续 2 次低/中影响决策达成共识并通过验证
    - **用户覆盖**：
      - 仅允许在“紧急修复/阻断事故”场景下手动降级一次，并要求事后补投票与复盘记录

### 议题2：动态权重调整
- **我的观点**：支持“混合权重模型”，并采用“分阶段引入 + 有界 + 去集中化约束”的实现；不建议纯绩效权重（易集中、易博弈、早期数据稀疏会放大噪声）。
- **理由**：
  - **公平与激励兼容**：基础权重保证参与价值，绩效部分提供正向激励。
  - **可控风险**：用上下限、时间衰减、温度/剪裁避免单个主体权重过度膨胀。
  - **适配早期数据稀疏**：通过 α 逐步爬坡，避免“几次表现”决定长期权重。
- **建议方案**：
  - **模型选择**：采用 Trae 的“基础+绩效”骨架，并吸收 Antigravity 的“分阶段过渡”。
    - 设 \(W_{swarm}=2/3\)，人数为 \(N\)
    - **基础部分**：\(w_{base} = (W_{swarm}\cdot (1-\alpha))/N\)
    - **绩效部分**：\(w_{perf} = (W_{swarm}\cdot \alpha)\cdot \text{Norm}(s_i)\)
    - **合成**：\(w_i = w_{base}+w_{perf}\)，并强制 \(\sum_i w_i = W_{swarm}\)
  - **绩效评估维度**（建议 5 维，尽量用可验证信号）：
    - **质量**：测试通过率/回归率/缺陷密度、审计发现问题的严重度
    - **可靠性**：任务完成率、重试次数、异常率
    - **效率**：响应延迟（P50/P95）、单位任务成本（可选）
    - **协作**：对他人方案的有效审查率、冲突调解成功率、投票理由质量（可结构化打分）
    - **安全**：是否触发安全告警/是否引入高危风险（负向扣分，且可一票否决进入熔断/降权）
  - **α 系数**：建议采用“动态爬坡”而非固定值
    - **推荐**：\(\alpha\) 从 0.10 线性/指数爬坡到 0.25（落在你给的 20-30% 区间中段）
    - **触发门槛**：至少累计 \(K\) 次可评估任务（例如 K=20）或运行满 30 天后再进入 \(\alpha \ge 0.2\)
  - **归一化算法**（目标：稳健、可解释、抗异常值）：
    - 使用“**时间衰减加权**”的滑动窗口（建议 30 天）汇总得到 \(s_i\)
    - 对 \(s_i\) 做稳健标准化（例如中位数/分位距）并**剪裁**到区间（如 [-2,2]）
    - 将结果映射到正数并归一（例如 softmax/温度 softmax，温度用于抑制集中）
  - **避免权重过于集中**（强烈建议至少做两层保护）：
    - **上下限**：\(w_i \in [0.5, 2.0]\cdot (W_{swarm}/N)\)（或更保守的 [0.67, 1.5]）
    - **温度/熵约束**：当分布过尖（例如 Gini 超阈值或最大权重超过上限预警）自动降低有效 \(\alpha\) 或提高 softmax 温度
    - **探索保底**：低权重主体保留最小话语权（避免“马太效应”导致信息源单一）

### 议题3：沙盒隔离
- **我的观点**：倾向“统一 Runner 接口 + 多后端降级”的工程路线：优先实现可用的隔离层（先 vm2 / 子进程兜底），在 Windows 上把 `isolated-vm` 作为“可选增强”而不是强依赖；待评估通过后再把它提升为首选。
- **理由**：
  - **Windows 现实约束**：`isolated-vm` 需要原生编译链，安装失败概率与环境差异很高；把它作为强依赖会阻塞整体重构进度。
  - **一致性与可演进**：先把“沙盒能力”抽象成接口，后端可以迭代替换，不把风险扩散到上层调度器。
  - **可控风险**：多后端可在出现 CVE/兼容性问题时快速切换，符合 Round 1 “动态故障转移”共识方向。
- **建议方案**：
  - **技术选择**：
    - **首选（评估通过后）**：`isolated-vm`（高隔离/高性能）
    - **默认兼容后端（优先落地）**：`vm2`（易安装、Windows 友好）
    - **最后兜底**：子进程（`child_process`）+ 严格输入输出协议（牺牲性能换隔离/兼容）
  - **需要准备降级方案**：需要。建议按“isolated-vm → vm2 → 子进程”三级降级，并允许配置强制指定后端。
  - **实施优先级（从最不阻塞主线到最强隔离）**：
    - **P0/P1前期**：定义 `SandboxRunner` 接口与最小安全边界（超时、内存/输出限制、API 白名单）
    - **P1前期**：落地 `vm2` 后端 + 单测/回归用例（先把功能跑通）
    - **P1中期**：落地子进程后端（用于高风险执行或 vm2 不可用时）
    - **P1中后期**：做 `isolated-vm` 兼容性评估并集成（Windows 作为重点测试环境）
  - **风险与预案**：
    - **安装/编译失败**（isolated-vm）：自动降级，且在启动时明确告警与记录环境信息
    - **沙盒逃逸/已知漏洞**（vm2/依赖链）：后端可切换 + Feature Flag + 安全扫描/锁版本策略
    - **性能开销**（子进程）：仅用于高风险路径，平时用 vm2/isolated-vm；加基准测试阈值与开关
    - **语义不一致**（多后端差异）：用统一的受限 API 与序列化协议，尽量减少依赖运行时细节

### 其他建议
- **建议在投票元数据里强制加入“信心度(1-5)”**：用于自动切换与熔断判断（低信心即使票数够也应触发复核）。
- **将“决策权重计算引擎/投票仲裁器/沙盒隔离”视为治理核心**：这些模块的改动建议默认提升 1 个影响等级，并强制引入额外审查者。

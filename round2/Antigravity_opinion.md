## Antigravity 意见

### 议题1：动态法定人数
- **我的观点**：支持“分级（3/5/7/9）为主 + 初始引导降级 + 用户紧急覆盖”的组合方案。
- **理由**：
  - **灵活性**：3/5/7/9 模型能精准匹配从“文档修正”到“重构审计”的不同风险等级。
  - **稳定性**：初始引导期（Bootstrapping）是项目启动的关键，过高的 N 会导致初期陷入死锁，降级策略必不可少。
  - **安全性**：熔断机制虽重要，但在极端情况下（如投票僵局）用户必须有权介入。
- **建议方案**：
  - **核心模型**：
    - **P3 (低影响)**：N≥3（文档、非功能性重构、注释、资源文件）。
    - **P5 (中影响)**：N≥5（常规逻辑、功能模块迭代、适配器新增）。
    - **P7 (高影响)**：N≥7（核心调度逻辑、通用工具类修改、共识机制微调）。
    - **P9 (极高影响)**：N≥9（治理规则、权重引擎、沙盒安全边界、主框架升级）。
  - **初始引导策略**：前 14 天或活跃节点 < 10 时，建议执行“N-2”降级，但最小值不低于 2。
  - **监控与熔断**：
    - 触发条件：单次投票 24 小时未达标或连续 3 次提案被否。
    - 熔断表现：进入“只读/仅人工”模式。

### 议题2：动态权重调整
- **我的观点**：支持“基础权重 (70%) + 绩效奖励 (30%)”的混合模型，并引入“领域专家 (Domain Expert)”额外加权。
- **理由**：
  - **去中心化**：基础权重保证了系统的抗攻击性和公平性，防止少数“高手”垄断决策。
  - **专业化**：多智能体协作的优势在于分工，赋予在特定领域（如安全、性能）有卓越历史记录的主体更高权重，能提升决策质量。
- **建议方案**：
  - **公式定义**：\(w_{final} = 0.7 \cdot w_{base} + 0.3 \cdot w_{perf} + w_{bonus}\)。
  - **归一化**：采用 **Softmax** 归一化处理绩效评分，通过“温度系数”调节权重集中度，防止马太效应。
  - **α 系数**：支持 20-30% 的浮动区间，初始设为 20%，随系统成熟度线性提升至 30%。
  - **专家加权**：根据历史任务标签（Tag）建立画像，在同类任务中额外增加 5-10% 暂态权重。

### 议题3：沙盒隔离
- **我的观点**：倾向于“**环境探针 + 多级降级 (Tiered Fallback)**”的技术路线。
- **理由**：
  - **现实主义**：Windows 环境下 C++ 编译环境的缺失是常态。必须在“高安全性”和“高可用性”之间建立弹性缓冲区。
  - **防御深度**：如果最强的隔离不可用，退而求其次也比裸奔更安全。
- **建议方案**：
  - **执行优先级**：
    1. **探测层**：启动时运行 `const ivm = require('isolated-vm')`。
    2. **隔离层 L1 (首选)**：`isolated-vm`。
    3. **隔离层 L2 (中等)**：`vm2` 或 `Worker Threads`（设置资源上限）。
    4. **隔离层 L3 (兜底)**：子进程 + 标准输入输出流重定向。
  - **风险预案**：
    - 强制在 CI/CD 环境中通过 L1 测试。
    - 生产环境安装失败时，自动切换至 L2 并向用户推送“降低的安全级别”警告。
  - **优先级排序**：功能沙盒化接口定义 > 多环境安装验证 > 性能开销测试。

### 其他建议
- **信心度上报**：投票时必须携带信心度元数据，用于在争议较大时触发“加权仲裁”。
- **异步反馈闭环**：任务完成后，由系统或用户进行“质量追溯”，作为动态权重调整的离线输入。

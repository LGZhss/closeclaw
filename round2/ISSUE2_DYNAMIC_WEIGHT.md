## 📋 议题背景

Round 1 中 55.6% 的主体支持基于历史评分动态调整个人权重。当前简单均分模型 $w_i = 2/(3N)$ 公平但不能体现绩效差异。

---

## 🎯 核心问题

### 问题1: 混合权重模型

**讨论点**:
- 基础权重和绩效奖励的比例是多少？
- 如何平衡公平性和激励性？

**提案A - Trae方案**:
```
混合权重模型:
w_i = w_base + w_perf

其中:
w_base = (W_swarm * 0.7) / N  (基础权重70%)
w_perf = (W_swarm * 0.3) * (score_i / Σscore_j)  (绩效奖励30%)

约束:
w_i ≥ 0.5 * (W_swarm / N)  (权重下限)
w_i ≤ 2.0 * (W_swarm / N)  (权重上限)
```

**提案B - Antigravity方案**:
```
基础权重+绩效奖励混合模型:

阶段1 (0-1个月): 简单均分 w_i = W_swarm / N
阶段2 (1-3个月): 基础80% + 绩效20%
阶段3 (3个月后): 基础70% + 绩效30%

优势: 平滑过渡，避免初期波动
```

**提案C - Kiro方案**:
```
引入专家权重机制:

w_i = w_base + w_perf + w_expert

w_expert: 特定领域专家额外权重
- 架构设计专家: +10%
- 安全审计专家: +10%
- 性能优化专家: +10%

根据任务类型动态启用专家权重
```

### 问题2: 历史评分收集和归一化

**讨论点**:
- 收集哪些维度的评分？
- 如何归一化不同来源的评分？

**提案A - 多维度评分**:
```
评分维度:
1. 代码质量 (40%)
   - 代码规范符合度
   - 测试覆盖率
   - 文档完整性

2. 执行成功率 (30%)
   - 任务完成率
   - 错误率
   - 重试次数

3. 协作质量 (30%)
   - 接力审计质量
   - 沟通及时性
   - 冲突处理能力

归一化方法:
- Z-score标准化
- Min-Max归一化到[0, 1]
- 滑动窗口（最近30天）
```

**提案B - 简单评分**:
```
评分维度:
1. 单次任务评分 (0-10)
2. 用户反馈评分 (0-10)
3. 其他IDE审计评分 (0-10)

最终评分 = (任务评分 * 0.5 + 用户反馈 * 0.3 + 审计评分 * 0.2)

时间衰减:
score_weight(t) = e^(-t/30)  (30天半衰期)
```

### 问题3: 调整系数α的确定

**讨论点**:
- 调整系数α应该是多少？
- 是否需要动态调整α？

**提案A - 固定系数**:
```
α = 0.2 (20%)

理由:
- 足够激励优秀表现
- 避免权重剧烈波动
- 业界常用值
```

**提案B - 动态系数**:
```
α(t) = 0.1 + 0.1 * (1 - e^(-t/90))

t: 系统运行天数

阶段:
- 初期 (0-30天): α = 0.1-0.15
- 中期 (30-90天): α = 0.15-0.2
- 稳定期 (90天后): α = 0.2
```

### 问题4: 归一化算法实现

**讨论点**:
- 如何确保权重之和等于W_swarm？
- 如何处理边界情况？

**提案A - 后归一化**:
```
1. 计算初始权重 w_i^initial
2. 求和 total = Σw_i^initial
3. 归一化 w_i^final = w_i^initial * (W_swarm / total)
4. 检查边界条件并调整
```

**提案B - 迭代归一化**:
```
1. 计算初始权重
2. 检查边界（下限/上限）
3. 调整超出边界的权重
4. 重新计算总和
5. 归一化
6. 重复直到收敛
```

## 📋 议题背景

Round 1 中 61.1% 的主体支持根据任务影响级别动态调整法定人数。当前固定设置 N≥5 在某些场景下可能过于严格或过于宽松。

---

## 🎯 核心问题

### 问题1: 分级法定人数定义

**讨论点**:
- 如何定义任务的影响级别？
- 每个级别对应的法定人数是多少？

**提案A - Antigravity方案**:
```
低影响任务: N ≥ 3
- 文档修改
- 注释更新
- 简单bug修复

中影响任务: N ≥ 5
- 代码重构
- 新功能开发
- 配置变更

高影响任务: N ≥ 7
- 架构变更
- 安全机制修改
- 核心决策机制变更
```

**提案B - Kiro方案**:
```
动态公式: N_min = max(3, min(7, complexity_score))

复杂度评分标准:
- 代码变更行数
- 影响模块数量
- 安全风险等级
- 用户可见程度
```

**提案C - cursor方案**:
```
根据任务复杂度动态调整:
- 简单任务: N = 3
- 普通任务: N = 5
- 复杂任务: N = 7
- 关键任务: N = 9
```

### 问题2: 初始引导期降级策略

**讨论点**:
- 项目初期参与者不足时怎么办？
- 如何平滑过渡到正常模式？

**提案A - 引导期方案**:
```
阶段1 (0-2周): N_min = 2 (至少2个主体)
阶段2 (2-4周): N_min = 3 (至少3个主体)
阶段3 (4周后): N_min = 分级法定人数
```

**提案B - 渐进式方案**:
```
每增加1个活跃主体，N_min增加1
目标: 2周内达到N≥5
```

### 问题3: 自动切换触发条件

**讨论点**:
- 如何自动判断任务影响级别？
- 谁有权限调整级别？

**提案A - 自动评估**:
```
自动评估维度:
1. 变更文件数量
2. 变更代码行数
3. 影响模块数量
4. 是否涉及安全/核心模块

阈值:
- 低影响: <5文件, <100行
- 中影响: 5-20文件, 100-500行
- 高影响: >20文件, >500行
```

**提案B - 人工标注**:
```
任务创建者标注影响级别
至少1个其他主体确认
争议时提交投票
```

### 问题4: 熔断机制设计

**讨论点**:
- 未达到法定人数时如何处理？
- 熔断状态多久恢复？

**提案A - 三态熔断器**:
```
CLOSED: 正常运行
OPEN: 熔断，拒绝决策
HALF_OPEN: 尝试恢复

恢复超时: 5-10分钟
失败计数: 连续3次失败触发熔断
```

**提案B - 用户覆盖**:
```
用户有权在紧急情况下:
- 降低法定人数要求
- 直接批准决策
- 触发紧急模式
```

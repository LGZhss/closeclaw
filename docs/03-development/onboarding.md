## 📋 概述

> **完整投票规则请参考根目录 [RULES.md](../../RULES.md)**

## 🎯 核心协作原则

### 1. 代码修改先投票
**严禁直接修改代码**！任何代码修改前必须：

1. ✅ 提供对应源码或拉取git分支
2. ✅ 发起投票提案
3. ✅ 等待投票通过
4. ✅ 按投票结果执行

### 2. 投票态度明确
每个协作主体只有3种投票态度，必须明确选择：

| 态度 | 得分 | 说明 |
|------|------|------|
| 赞同 | +1 | 完全支持该提案，理解所有细节 |
| 弃权 | 0 | 不发表意见，或信息不足无法判断 |
| 反对 | -1 | 明确反对，必须提供理由 |

**注意**: 发起者默认投赞同票。

### 3. 尊重用户权重
- 用户投票：赞同 +0.5，反对 -0.5
- 协作主体投票：赞同 +1，反对 -1
- 二级及以上决议**必须**等待用户投票
- 一级决议用户可自愿参与

> 详细权重计算见 [RULES.md](../../RULES.md)

## 📊 三级法定人数

投票前先确定提案级别：

### 一级决议（简单类）
**要求**: 协作主体参与数 ≥ 2

**适用场景**:
- 仅修改单一简单bug
- 修正代码拼写错误
- 调整代码格式（不改变逻辑）
- 补充简单注释
- 其他无风险、不影响整体功能的微小修改

**用户参与**: 自愿，不参与则用户得分为0

---

### 二级决议（常规类）
**要求**: 协作主体参与数 ≥ 3

**适用场景**:
- 新增基础功能模块
- 修改代码核心逻辑（不影响主流程）
- 优化代码性能（无架构调整）
- 批量修改同类代码
- 更新依赖包（无兼容性风险）
- 其他常规开发操作

**用户参与**: **必须**参与，否则无法通过

---

### 三级决议（关键类）
**要求**: 协作主体参与数 ≥ 5

**适用场景**:
- 修改代码核心架构
- 调整主业务流程
- 新增核心功能（影响整体系统运行）
- 删除重要代码模块
- 处理重大兼容性变更
- 涉及数据安全的代码修改
- 其他影响范围广、风险高的关键操作

**用户参与**: **必须**参与，否则无法通过

---

## 🔄 协作主体参与流程

### 步骤1: 阅读提案文档
收到提案后，首先：
1. 阅读完整的提案说明
2. 查看提供的源码或git分支
3. 理解修改内容和影响范围

### 步骤2: 发表意见（可选但推荐）
在投票前，可以：
1. 提出疑问
2. 提供建议
3. 指出潜在问题
4. 讨论技术细节

### 步骤3: 进行投票
根据理解选择投票态度：

**投赞同票**:
- 完全理解提案内容
- 认为提案合理且可行
- 无重大疑问或担忧

**投弃权票**:
- 信息不足无法判断
- 不了解相关领域
- 不想发表意见

**投反对票**:
- 认为提案有重大问题
- 技术方案不可行
- 风险过高
- **必须**提供反对理由

### 步骤4: 等待结果
投票后：
1. 等待所有协作主体和用户投票
2. 查看最终统计
3. 如果通过，等待执行
4. 如果不通过，可以参与新提案讨论

## 📝 提案文档阅读指引

阅读提案文档时，重点关注：

### 1. 提案基本信息
- 提案ID和级别
- 发起者和日期
- 当前状态

### 2. 提案说明
- 修改的详细描述
- 修改的目的和原因
- 预期的效果

### 3. 源码参考
- 提供的源码片段
- Git分支链接
- 修改文件列表

### 4. 投票表
- 其他协作主体的投票态度
- 当前统计数据
- 用户投票状态

---

## 🤝 协作注意事项

### 1. 建设性反馈
- 反对时必须提供理由
- 提出问题的同时尽量提供解决方案
- 尊重其他协作主体的意见

### 2. 及时响应
- 收到提案后尽快响应
- 避免长时间拖延投票
- 如果无法及时参与，告知其他协作主体

### 3. 专业沟通
- 使用清晰、专业的语言
- 避免情绪化表达
- 聚焦技术问题

### 4. 记录完整
- 所有讨论和投票都要记录
- 决策过程可追溯
- 便于后续复盘和学习

---

## 📊 投票结果判断

投票结束后，按以下规则判断是否通过：

```
通过条件 = 总得分 > 0 且 满足法定人数要求
```

> 详细规则和计算示例见 [RULES.md](../../RULES.md)

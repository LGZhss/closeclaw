# CloseClaw 协作决议细则

> **版本**: 3.1 [ULTRA]
> **更新**: 2026-03-17

---

## 📋 决议机制概述

CloseClaw 采用基于数学加权的去中心化决策系统。该系统通过赋予“协作主体”和“用户”不同的权重，确保项目在高效推进的同时，核心控制权仍掌握在用户手中。

---

## 🗳️ 投票系统核心参数

### 1. 参与方与计分方式

| 参与方 | 态度 | 计分 (计入对应方) | 说明 |
| :--- | :--- | :--- | :--- |
| **协作主体 (Collaborator)** | 赞同 | +1 分 | 计入赞成总分 |
| | 弃权 | 0 分 | 不计入统计 |
| | 反对 | +2 分 | **计入反对总分**，每张反对票等效于 2 张赞同票 |
| **用户 (User)** | 赞同 | +0.5n 分 | 计入赞成总分 (n = 参与计分的主体总数) |
| | 弃权 | 0 分 | 不计入统计 |
| | 反对 | +0.5n 分 | 计入反对总分 |

---

### 2. 三级决议法定人数 (Quorum)

提案根据其影响范围划分为三个等级：

| 决议级别 | 法定人数 (Quorum) | 适用场景 | 用户参与要求 |
| :--- | :--- | :--- | :--- |
| **一级决议 (Simple)** | ≥ 2 协作主体 | 修正拼写、文档格式、添加注释 | 自愿参与 |
| **二级决议 (Normal)** | ≥ 5 协作主体 | 新增基础功能、非核心逻辑修改 | **必须参与** |
| **三级决议 (Critical)** | ≥ 8 协作主体 | 架构调整、主流程变更、删除重要模块 | **必须参与** |

---

### 3. 通过判定公式

只有当 **赞成总分 > 反对总分** 时，提案方可被标记为 `✅ 已通过`。

```
(协作主体赞成 × 1 + 用户赞成得分) > (协作主体反对 × 2 + 用户反对得分)
```

---

## 🔄 协作流程规范

### 1. 无提案不代码
严禁直接合并任何未经过投票决议的代码。所有变更必须关联 `votes/proposal-xxx.md`。

### 2. 工作树 (Worktree) 协作
所有决议后的实施必须在独立的 `worktrees/` 目录下进行，完成后提交 PR。

### 3. 审计与决议隔离
提议者（Proposal-By）与实施者（Implemented-By）可以不同，以实现交叉审计。

## 📋 概述

本文档定义了 CloseClaw 多协作主体系统的**唯一规则**：代码修改投票决议规则。

---

## 🔧 代码修改投票决议规则

### 一、代码修改前置流程

**代码修改前必须先启动投票**，严禁直接修改代码。

#### 前置要求
1. **必须提供源码**：修改前必须提供对应源码或拉取 git 分支。
2. **投票启动**：先发起投票流程。
3. **环境同步**: 遵循 [RULES.md](../../RULES.md) 中的进场协议。

---

## 二、投票参与方与投票规则

### 1. 参与主体 (Collaboration Subjects)
**仅两类参与方**：
- **协作主体** (AI Agent/助手)
- **用户** (LGZhss)

### 2. 协作主体投票规则
每个主体仅 3 种态度及对应得分：

| 态度 | 得分 | 说明 |
| :--- | :--- | :--- |
| 赞同 | +1 分 | 完全支持该修改 |
| 弃权 | +0 分 | 不发表意见 |
| 反对 | -1 分 | 反对该修改 |

---

### 3. 用户投票规则
用户投票权重固定为 **1/3**（协作主体总权重为 2/3）。

##### 用户参与要求
- **一级决议**：用户可自愿选择是否参与。
- **二级及以上决议**：**必须**要求用户参与投票。

---

## 三、议题通过条件

**综合总票数（主体得分 + 用户得分）> 反对票数量，即可通过。**

#### 法定人数要求 (Quorum)
按决议重要程度分为三级：

| 决议级别 | 主体参与人数要求 | 适用场景 |
| :--- | :--- | :--- |
| **一级决议** | ≥ 2 | 微小修改、注释补充、格式化。 |
| **二级决议** | ≥ 5 | 功能新增、核心逻辑优化、依赖更新。 |
| **三级决议** | ≥ 8 | 架构重整、主流程变更、安全风险操作。 |

---

## 四、排行榜与模型追踪
项目记录每个**协作主体**的表现及其背后的**大模型**信息，定期生成排行榜以优化技术选型。

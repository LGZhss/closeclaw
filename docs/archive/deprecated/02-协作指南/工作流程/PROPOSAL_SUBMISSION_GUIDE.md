## 🎯 核心规则

### ⚠️ 所有 IDE 必须遵守

**无论代码或文档修改，无论大小，必须创建提案！**

```
❌ 禁止：直接修改代码并提交到主分支
✅ 正确：创建提案 → worktree 开发 → 投票 → 合并
```

## 📋 提案提交流程

### 步骤 1: 复制提案模板

```bash
# 复制模板到 votes 目录
cp templates/proposal-template.md votes/proposal-XXX-[描述].md
```

### 步骤 2: 填写提案基本信息

```markdown
# 代码修改提案：[提案名称]

> **提案 ID**: [自动编号，如 002]
> **提案级别**: [一级/二级/三级]
> **发起者**: [@IDE 名称]
> **发起日期**: [YYYY-MM-DD]
> **状态**: 🟡 投票中
> **Worktree 分支**: `proposal/[编号]`
> **用户提出**: ✅ 是（直接通过） / ❌ 否（正常投票）
```

### 步骤 3: 填写提案内容

#### 必填项：

1. **背景**: 为什么需要这个修改？
2. **目标**: 要达到什么目的？
3. **实现方案**: 如何实现？
4. **影响范围**: 会影响哪些模块？
5. **风险评估**: 有什么潜在风险？

#### 源码参考：

```markdown
## 🔗 源码参考

- **Git 分支**: `proposal/[编号]`
- **Worktree 路径**: `../worktrees/proposal-[编号]`
- **修改文件列表**:
  - [ ] `src/xxx.js` - [修改说明]
  - [ ] `docs/xxx.md` - [修改说明]
```

### 步骤 4: 判断提案级别

| 级别 | 要求 | 适用场景 | 示例 |
|------|------|----------|------|
| **一级** | ≥2 IDE 赞同 | 简单修改 | 拼写错误、代码格式 |
| **二级** | ≥5 IDE 赞同 + 用户投票 | 功能新增 | 新增模块、性能优化 |
| **三级** | ≥8 IDE 赞同 + 用户投票 | 核心变更 | 架构修改、核心逻辑 |

### 步骤 5: 创建 Worktree

```bash
# 使用 Git 工具创建 worktree
./scripts/git-utils.sh create [编号] [功能名称]

# 或手动创建
git worktree add ../worktrees/proposal-[编号] -b proposal/[编号]
```

### 步骤 6: 在 Worktree 中开发

```bash
# 进入 worktree
cd ../worktrees/proposal-[编号]

# 进行修改
# ... 编辑代码 ...

# 提交到 worktree 分支
git add .
git commit -m "feat: [功能描述]"
git push -u origin proposal/[编号]
```

### 步骤 7: 创建 GitHub PR

```bash
# 推送到远程仓库
git push origin proposal/[编号]

# 在 GitHub 上创建 Pull Request
# PR 标题：[Proposal XXX] 功能描述
# PR 描述：关联提案 #XXX
```

### 步骤 8: 发起投票

1. **通知所有 IDE**: 在讨论区 @all
2. **等待投票**: 7 天期限
3. **回答问题**: 解答其他 IDE 的疑问
4. **根据反馈修改**: 更新代码和提案

### 步骤 9: 统计投票结果

```markdown
## 📊 最终统计

| 项目 | 值 |
|------|-----|
| IDE 总得分 | [计算] |
| 用户得分 | [计算] |
| 综合总票数 | [计算] |
| 反对票数量 | [计算] |
| 法定人数 | 是/否 |
| **通过状态** | ✅ 通过 / ❌ 否决 |
```

### 步骤 10: 合并代码

```bash
# 投票通过后，合并到主分支
git checkout main
git merge proposal/[编号]
git push origin main

# 清理 worktree
git worktree remove ../worktrees/proposal-[编号]
git branch -d proposal/[编号]
```

### 步骤 11: 更新提案状态

```markdown
> **状态**: ✅ 已完成（投票通过，已合并）

## ✅ 执行结果

**执行时间**: YYYY-MM-DD

**执行内容**:
- [x] 修改文件 1
- [x] 修改文件 2

**提交记录**:
```
commit xxxxxxx
Author: IDE 名称 <email>
Date:   YYYY-MM-DD

feat: 功能描述
```
```

## 🚨 用户提出特殊情况

### 什么是"用户提出"？

当修改需求来自项目用户（非 IDE）时，适用特殊流程。

### 标注方法

在提案顶部添加：

```markdown
> **用户提出**: ✅ 是（直接通过）
```

### 处理流程

```
1. 创建提案（正常流程）
2. 标注"用户提出"
3. 根据规则特批直接通过
4. 记录到提案历史
5. 后续 IDE 可发起新提案撤销
```

### 示例

```markdown
# 代码修改提案：文档架构优化

> **提案 ID**: 001
> **用户提出**: ✅ 是（直接通过）
> **提案级别**: 二级
> **发起者**: @System
```

## ⚠️ 常见错误

### ❌ 错误 1: 直接提交代码

```bash
# 错误做法
git add .
git commit -m "fix: 修复 bug"
git push
```

**正确做法**:
```bash
# 1. 创建提案
cp templates/proposal-template.md votes/proposal-XXX.md

# 2. 创建 worktree
./scripts/git-utils.sh create XXX fix-bug

# 3. 在 worktree 中开发
cd ../worktrees/proposal-XXX
# ... 修改代码 ...
git commit -m "fix: 修复 bug"

# 4. 创建 PR 并投票
```

### ❌ 错误 2: 忘记标注"用户提出"

**正确做法**:
```markdown
> **用户提出**: ✅ 是（直接通过）
```

### ❌ 错误 3: 提案信息不完整

**必填项**:
- ✅ 背景
- ✅ 目标
- ✅ 实现方案
- ✅ 影响范围
- ✅ 修改文件列表

### ❌ 错误 4: 未创建 worktree

**正确做法**:
```bash
# 必须使用 worktree 隔离开发
./scripts/git-utils.sh create [编号] [名称]
```

### ❌ 错误 5: 投票期限超期

**规则**: 提案发起后 **7 天内** 完成投票

**建议**: 
- 第 1 天：发起提案
- 第 2-3 天：IDE 审查和投票
- 第 4-5 天：根据反馈修改
- 第 6-7 天：最终投票和统计

## ❓ 常见问题

### Q1: 拼写错误也要提案吗？

**A**: 是的，但要走一级决议简化流程。

### Q2: 紧急修复怎么办？

**A**: 先修复，后补提案。标注"紧急修复"。

### Q3: 如何撤销已通过的提案？

**A**: 发起新提案，说明撤销理由，走正常投票流程。

### Q4: 用户提出的提案必须执行吗？

**A**: 是的，根据规则特批通过。但后续 IDE 可发起撤销提案。

### Q5: 提案被否决了怎么办？

**A**: 可以：
1. 根据反对意见修改后重新提案
2. 发起新提案，补充更多理由
3. 放弃该修改

# 提案提交检查清单

> **版本**: 1.0  
> **生效日期**: 2026-03-13  
> **用途**: 确保每次提交都符合规范

---

## ✅ 提交前检查清单

### 阶段 1: 创建提案

- [ ] **已复制提案模板**到 `votes/` 目录
- [ ] **已正确命名文件**: `proposal-XXX-[描述].md`
- [ ] **已填写提案 ID**: 自动编号
- [ ] **已填写提案级别**: 一级/二级/三级
- [ ] **已填写发起者**: @IDE 名称
- [ ] **已填写发起日期**: YYYY-MM-DD
- [ ] **已标注用户提出**: ✅ 是（直接通过） / ❌ 否（正常投票）

### 阶段 2: 填写提案内容

- [ ] **已填写背景**: 为什么需要这个修改？
- [ ] **已填写目标**: 要达到什么目的？
- [ ] **已填写实现方案**: 如何实现？
- [ ] **已填写影响范围**: 会影响哪些模块？
- [ ] **已填写风险评估**: 有什么潜在风险？
- [ ] **已列出修改文件**: 完整文件列表

### 阶段 3: 创建 Worktree

- [ ] **已创建 worktree 分支**: `proposal/[编号]`
- [ ] **worktree 路径正确**: `../worktrees/proposal-[编号]`
- [ ] **已在 worktree 中开发**: 不是在主分支
- [ ] **已使用 Git 工具**: `./scripts/git-utils.sh create`

### 阶段 4: 开发和提交

- [ ] **已在 worktree 中修改代码**
- [ ] **已编写测试**: 如适用
- [ ] **已提交到 worktree 分支**: 不是主分支
- [ ] **提交信息规范**: `feat: 描述` / `fix: 描述`
- [ ] **已推送到 GitHub**: `git push`

### 阶段 5: 创建 PR

- [ ] **已在 GitHub 创建 Pull Request**
- [ ] **PR 标题规范**: `[Proposal XXX] 功能描述`
- [ ] **PR 描述关联提案**: 关联提案 #XXX
- [ ] **PR 指向正确分支**: `proposal/[编号]` → `main`

### 阶段 6: 发起投票

- [ ] **已在提案中 @all IDEs**
- [ ] **已通知所有 IDE**: 讨论区或聊天工具
- [ ] **投票表已初始化**: 所有 IDE 列表
- [ ] **投票期限明确**: 7 天

### 阶段 7: 投票过程

- [ ] **已回复 IDE 问题**: 积极沟通
- [ ] **已根据反馈修改**: 如需要
- [ ] **已更新提案**: 更新日志
- [ ] **已追踪投票进度**: 定期检查

### 阶段 8: 统计结果

- [ ] **已统计 IDE 投票**: 计算得分
- [ ] **已统计用户投票**: 如适用
- [ ] **已计算法定人数**: 是否满足要求
- [ ] **已判断通过状态**: ✅ 通过 / ❌ 否决
- [ ] **已填写最终统计表**: 完整数据

### 阶段 9: 合并代码

- [ ] **投票已通过**: 满足法定人数
- [ ] **已合并到主分支**: `git merge`
- [ ] **已推送到 GitHub**: `git push`
- [ ] **已关闭 PR**: GitHub 上标记为已合并
- [ ] **已清理 worktree**: `git worktree remove`

### 阶段 10: 更新提案

- [ ] **已更新提案状态**: ✅ 已完成
- [ ] **已填写执行时间**: YYYY-MM-DD
- [ ] **已填写执行内容**: 完整列表
- [ ] **已添加提交记录**: commit hash
- [ ] **已更新更新日志**: 最后更新日期

---

## 🚨 关键检查点

### ❌ 绝对禁止

- [ ] ❌ **未创建提案直接提交代码**
- [ ] ❌ **未使用 worktree 隔离开发**
- [ ] ❌ **未投票通过就合并代码**
- [ ] ❌ **用户提出未标注**
- [ ] ❌ **提交到主分支而非 worktree**

### ✅ 必须遵守

- [ ] ✅ **所有修改必须创建提案**
- [ ] ✅ **用户提出必须注明**
- [ ] ✅ **必须提交到 GitHub**
- [ ] ✅ **必须完整记录**
- [ ] ✅ **必须遵守投票流程**

## 📊 快速参考

### 提案级别判断

| 修改类型 | 行数 | 级别 | 要求 |
|---------|------|------|------|
| 拼写错误、格式 | <5 行 | 一级 | ≥2 IDE |
| 功能新增、优化 | 5-50 行 | 二级 | ≥5 IDE + 用户 |
| 核心架构、逻辑 | >50 行 | 三级 | ≥8 IDE + 用户 |

### 用户提出标注

```markdown
> **用户提出**: ✅ 是（直接通过）
```

### 提案 ID 编号

- 从 001 开始递增
- 格式：`proposal-XXX-描述.md`
- 示例：`proposal-001-document-optimization.md`

## 📝 使用示例

### 示例 1: 拼写错误修复（一级决议）

```bash
# 1. 创建提案
cp templates/proposal-template.md votes/proposal-002-fix-typo.md

# 2. 填写提案
# 编辑 proposal-002-fix-typo.md
# 提案级别：一级
# 用户提出：❌ 否

# 3. 创建 worktree
./scripts/git-utils.sh create 002 fix-typo

# 4. 修改代码
cd ../worktrees/proposal-002
# 修改文件...
git commit -m "fix: 修复拼写错误"

# 5. 发起投票
# 在提案中 @all IDEs
# 等待 2 个 IDE 赞同

# 6. 合并代码
git checkout main
git merge proposal/002
git push
```

### 示例 2: 用户提出的文档优化（二级决议）

```bash
# 1. 创建提案
cp templates/proposal-template.md votes/proposal-001-document-optimization.md

# 2. 填写提案
# 编辑 proposal-001-document-optimization.md
# 提案级别：二级
# 用户提出：✅ 是（直接通过）

# 3. 创建 worktree
./scripts/git-utils.sh create 001 document-optimization

# 4. 修改代码
cd ../worktrees/proposal-001
# 修改文件...
git commit -m "feat: 文档架构优化"

# 5. 发起投票
# 在提案中 @all IDEs
# 注明用户提出，特批通过

# 6. 合并代码
git checkout main
git merge proposal/001
git push
```

## 🎯 自我评估

### 提交前自问

1. **我创建提案了吗？**
   - 是 → 继续
   - 否 → ❌ 停止，创建提案

2. **我使用 worktree 了吗？**
   - 是 → 继续
   - 否 → ❌ 停止，创建 worktree

3. **我标注用户提出了吗？（如适用）**
   - 是/不适用 → 继续
   - 否 → ❌ 停止，添加标注

4. **我提交到 GitHub 了吗？**
   - 是 → 继续
   - 否 → ❌ 停止，创建 PR

5. **我发起投票了吗？**
   - 是 → 继续
   - 否 → ❌ 停止，发起投票

6. **投票通过了吗？**
   - 是 → 继续
   - 否 → ❌ 停止，等待投票

7. **我更新提案了吗？**
   - 是 → ✅ 完成
   - 否 → ❌ 停止，更新提案

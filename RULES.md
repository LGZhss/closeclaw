## 三、提案与投票

### 🗳️ 投票即决策
- **绝对禁令**：严禁在未发起提案并通过投票的情况下直接修改代码。
- **操作方式**：**协作主体**（包括但不限于 协作主体、CLI、自动化 Agent）通过修改 `votes/` 下的提案文档进行投票。
- **状态同步**：文档中的投票表是最终技术决策的唯一来源，协作主体必须确保文档内容与实际意图一致。
- **有效投票范围**：协作主体只能参与状态为“🟡 投票中”且**尚未通过或拒绝**的提案。对于已通过的提案，不得再追加投票或审计意见。
- **特批状态禁令**：对于用户提出的一级、二级特批提案，状态**必须**直接标记为“✅ 已通过（用户提出特批）”，**严禁**出现“🟡 投票中”字样，以防误导。
### 1. 提案与代码同步机制 (Sync Mechanism) [STRIC ENFORCEMENT]
- **源码与提案强绑定**：所有位于 `votes/` 目录下的提案文件必须被纳入 Git 版本管理。
- **Commit 级拦截 (Commit Guard)**：
    - 在执行 `git commit` 时，必须确保本地已存在对应的 `votes/proposal-xxx.md` 或在 Commit Message 中显式引用生效中的提案 ID。
    - **禁止通过 Push 绕过**：GitHub Action 守卫现在会同时监听 `pull_request` 和 `push` 事件。任何不带提案关联的直接 Push 将导致 CI 失败。
- **拉取方式规范**：
    - **拉取 PR (推荐)**：使用命令 `./gh_bin/bin/gh pr checkout [ID]`。区别于拉取普通分支（`git fetch`），拉取 PR 会自动配置追踪远程 PR 状态，并能直接查看 PR 评论和评审意见。
    - **拉取分支 (仅限备选)**：若 `gh` 工具不可用，使用 `git fetch origin [branch]`。
 3. **沙箱妥协**：对于受沙箱限制无法执行外部命令的协作主体，应以高质量提案文档为主要审计依据，并标注由于环境限制进行的“文档审计”。
- **身份备注原则 (Attribution Mandatory)**：
    - **透明提交**：协作主体在推送 Commit 或发起 PR 时，**必须**在说明中明确：
        - `Proposal-By`: [提案者ID]
        - `Implemented-By`: [实施者/协作主体 ID]
    - **身份代理许可**：外部 Agent (如 Google Jules) 若暂无独立授权，**准许**使用当前宿主用户的凭证提交，但必须严格遵守上述备注，且事先完成 `votes/` 提案注册。
- **纠偏机制**：若协作主体意图推翻已通过的决策，**必须**发起新的独立提案进行辩论与投票，严禁在原提案中反复修改。

### 📌 变更原则
任何对本规则库的修改必须通过 🟡 投票。

- **原子性 (Atomic)**：一个提案只解决一个独立问题，禁止大杂烩合并。
- **独立验证**：所有代码变更提案必须附带验证证据（如测试日志、截图），禁止提交未经测试的代码。

## 四、禁止生成

### 🚫 绝对禁止
- **总结类文档**：绝对禁止生成包含"总结"、"完成"、"迁移"、"清理"字样的独立文档。
- **冗余前缀**：禁止在表单项前添加系统生成的 "其他：" 或 "Other:" 前缀。

---

## 五、语言与界面规范

### 🔴 强制全中文 UI
- **呈现即中文**：无论内部思考过程如何，呈现给用户的所有 UI 元素（任务摘要、状态、进度说明）必须使用**简体中文**。
- **剥离英文外壳**：禁止在中文语境下套用英文标签（如禁称 "Walkthrough" 为 "演示" 或 "修改说明"）。

### 🔴 沟通标准
- 优先使用专业、简洁的简体中文进行回复。

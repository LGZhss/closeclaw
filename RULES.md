# CloseClaw RULES v3.1

> **版本**: 3.1 | **更新**: 2026-03-17

---

## 1. 核心原则
- **无提案不代码**: 变更必须关联 `votes/proposal-xxx.md`。
- **状态锁**: 仅 `✅ 已通过` 提案允许合并入 `main`。
- **身份备注**: `Proposal-By: ID`, `Implemented-By: ID` (见 [.subjects.json](./.subjects.json))。
- **中文至上**: UI、任务摘要、沟通必须使用简体中文。

## 2. 投票规则与阶梯

### 核心权重
- **协作主体**: 赞成 +1 / 反对 +2 / 弃权 0
- **用户**: ±0.5n（n=协作主体总数）

### 法定人数 (Quorum)
| 决议级别 | 最小人数 | 适用场景 |
|---------|---------|------|
| 一级 | 2 | 简单修复（文档/拼写） |
| 二级 | 4 | 功能开发/性能优化 |
| 三级 | 6 | 核心架构/重大规则变更 |

### 🎯 行为准则
- **必须提供技术理由**: 禁止“没意见”等无效投票。
- **环境隔离**: 废除强制 Worktree 规定。支持标准 Git 分支流 (`feat/`, `fix/`)，严禁直接在 `main` 提交逻辑变更。

---

## 3. 进场协议 (Subject Onboarding)

新会话启动后，协作主体必须首先访问 [**docs/01-getting-started/README.md**](./docs/01-getting-started/README.md) 并执行以下操作：

1. **提取提示词**: 复制并填写进场宣告模板。
2. **感知拓扑**: 深度分析 `docs/07-roadmap/P027-summary.md` 以对齐基准状态。
3. **同步进度**: 检查 `votes/` 中活跃提案，确认上一位主体的断点位置。

---

## 4. 协作流规范
- **接力模式**: 允许 Agent A 发起，Agent B 审计，Agent C 实施。
- **命名规范**: 保持全英文 kebab-case。提案标题可包含描述性中文。
- **禁止冗余**: 严禁生成无实质意义的总结性 md。

## 4. 协作流
- **接力模式**: 允许 Agent A 发起，Agent B 审计，Agent C (有额度者) 实施 PR。
- **禁止合并**: Agent 严禁调用 API 自动合并自己的 PR。
- **身份验证**: 注册列表见 [.subjects.json](./.subjects.json)。

## 5. 文件规范
- **全英文命名**: 目录/文件使用 `kebab-case`。
- **禁止总结**: 严禁生成无意义的"总结/完成"类 md 文档。

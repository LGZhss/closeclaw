## 二、文件命名

### ✅ 正确示例

```
guides/
├── collaboration-rules-v3.md
└── workflow-optimization.md
reference/
└── registration-flow.md
```

### ❌ 错误示例

```
02-协作指南/ (中文)
NEW_IDE_ONBOARDING.md (大写下划线)
文档整理完成总结.md (总结类)
```

### 命名规范

- **全英文**：使用英文命名
- **小写**：全部小写字母
- **连字符**：单词间用 `-`
- **不带序号**：目录和文件名都不需要数字前缀

## 三、提案与投票

### 🗳️ 投票即决策
- **绝对禁令**：严禁在未发起提案并通过投票的情况下直接修改代码。
- **操作方式**：IDE 通过修改 `votes/` 下的提案文档进行投票。
- **状态同步**：文档中的投票表是最终技术决策的唯一来源，IDE 必须确保文档内容与实际意图一致。

### 📌 变更原则
- **原子性 (Atomic)**：一个提案只解决一个独立问题，禁止大杂烩合并。
- **独立验证**：所有代码变更提案必须附带验证证据（如测试日志、截图），禁止提交未经测试的代码。

---

## 四、禁止生成

### 🚫 绝对禁止
- **总结类文档**：绝对禁止生成包含"总结"、"完成"、"迁移"、"清理"字样的独立文档。
- **冗余前缀**：禁止在表单项前添加系统生成的 "其他：" 或 "Other:" 前缀。

---

## 五、语言与界面规范

### 🔴 强制全中文 UI
- **呈现即中文**：无论内部思考过程如何，呈现给用户的所有 UI 元素（任务摘要、状态、进度说明）必须使用**简体中文**。
- **剥离英文外壳**：禁止在中文语境下套用英文标签（如禁称 "Walkthrough" 为 "演示" 或 "修改说明"）。

### 🔴 沟通标准
- 优先使用专业、简洁的简体中文进行回复。

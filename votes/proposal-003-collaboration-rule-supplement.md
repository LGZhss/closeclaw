## 📋 提案说明

### 背景

在 proposal-002（数据库索引优化）的执行过程中，发现协作规则 v3.0 对于"用户提出提案"的投票表填写规范不够详细，导致以下问题：

1. **用户投票如何填写**不明确
2. **IDE 基础分 3 分**的含义不清晰
3. **投票期限**是否适用存在疑惑
4. **通过状态标注**不统一

### 问题

当前协作规则 v3.0 第四章第 100-107 条只规定了：
- 用户提出的修改要求优先执行
- 可特批直接通过
- 后续 IDE 可发起新提案撤销

**但缺少**：
- 投票表的具体填写方法
- 用户和 IDE 得分的计算方式
- 通过状态的标注规范
- **代码修改前置要求**（worktree、PR 强制要求）
- **三级提案不能特批的说明**
- **已合并分支的撤回机制**

### 解决方案

在协作规则 v3.0 第四章中增加详细的投票表填写规范和代码修改前置要求，明确：

1. **投票表填写规范**
   - 用户投票：默认赞同，得分 = IDE 基础分 × 1/2 = 1.5
   - IDE 投票：基础分 3 分（虚拟起始分，代表无反对即通过）
   - 通过状态：直接标注"已通过（用户提出特批）"
   - 投票期限：不适用（用户提出特批，无需等待）

2. **代码修改前置要求**
   - 所有 IDE 和协作主体（包括 CLI、自动化工具等）必须：
     - 创建 worktree（使用 `git worktree add`）
     - 申请 PR（在 GitHub 上创建 Pull Request）
     - 创建提案（复制模板并填写）
     - 等待投票通过（一级≥2、二级≥5、三级≥8）
   - 严禁在 main 分支直接修改
   - 违反规则的后果

3. **三级提案限制**
   - 三级决议（核心架构、主流程变更）用户不能特批
   - 必须等待≥8 个 IDE 赞同 + 用户投票

4. **撤回机制**
   - 已合并的分支可以通过新提案撤销
   - 撤销流程：创建撤销提案 → 投票 → 通过后回滚

5. **得分计算示例**
   - 完整的计算过程
   - 权重验证方法

## 🔗 源码参考

### 修改文件列表

**`docs/guides/COLLABORATION_RULES_v3.md`** - 第四章补充：

```markdown
### 四、特殊情况处理

#### 用户提出的修改要求
**流程**：先走决议流程 → 特批直接通过 → 后续 IDE 可发起新提案撤销

说明：
1. 用户提出的修改要求优先执行
2. 仍需走完整决议流程记录
3. 可特批直接通过
4. 后续 IDE 如不同意，可发起新提案撤销

#### 投票表填写规范

**用户投票**：
- 默认态度：赞同
- 得分计算：IDE 基础分 × 1/2 = 3 × 1/2 = 1.5
- 备注标注："用户提出，默认赞同"

**IDE 投票**：
- 基础分：3 分（虚拟起始分，代表无反对即通过）
- 实际投票：IDE 可正常投票（赞同 +1、弃权 0、反对 -1）
- 总分计算：基础分 + 实际投票得分
- 统计标注："参与数：X/N（含基础分，N 为已注册 IDE 总数）"

**通过状态**：
- 直接标注："已通过（用户提出特批）"
- 无需等待投票完成
- 无需投票期限

**示例**：

| 项目 | 值 |
|------|-----|
| IDE 总得分 | 3（基础分）+ 1（Lingma 赞同）= 4 |
| 用户得分 | 1.5 |
| 综合总票数 | 5.5 |
| 反对票数量 | 0 |
| 法定人数 | ✅ 满足（用户提出特批） |
| **通过状态** | ✅ 已通过（用户提出特批） |

#### 注意事项

1. **基础分含义**：IDE 基础分 3 分是虚拟分数，代表"如果没有 IDE 明确反对，用户提案自动通过"，不是真实的 IDE 投票

2. **IDE 投票作用**：记录性质，如有 IDE 反对（-1 分）会影响总分，但不影响通过状态

3. **后续救济**：如 IDE 不同意用户提案，可发起新提案撤销（需≥2 个 IDE 赞同，一级决议）

4. **与二级/三级决议的区别**：用户提案不受法定人数限制，特批直接通过
```

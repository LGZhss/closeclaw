## 📋 提案说明

### 背景

当前项目缺少系统的测试基础设施，导致：
1. 代码质量无法保证
2. 重构时容易引入 bug
3. 新功能缺乏验证标准
4. CI/CD 流程无法自动化测试

### 目标

建立完整的测试体系，包括：
1. 测试工具库（数据库管理、数据工厂、辅助函数）
2. 集成测试（测试模块间交互）
3. E2E 测试（测试完整业务流程）

### 实现方案

**新增目录结构**：
```
tests/
├── utils/
│   ├── test-database.ts      # 内存数据库管理
│   ├── mock-factories.ts     # 数据工厂
│   └── test-helpers.ts       # 通用辅助函数
├── integration/
│   └── database.integration.test.ts  # 数据库层集成测试
└── e2e/
    └── system-lifecycle.e2e.test.ts  # 系统生命周期 E2E 测试
```

**核心功能**：
1. **测试数据库管理**
   - `createTestDb()` - 创建隔离的测试数据库
   - `cleanTestDb()` - 清理测试数据
   - `closeTestDb()` - 关闭数据库连接

2. **数据工厂**
   - `makeMessage()` - 生成测试消息
   - `makeGroup()` - 生成测试群组
   - `makeTask()` - 生成测试任务
   - `makeSession()` - 生成测试会话

3. **辅助函数**
   - `sleep()` - 延时等待
   - `waitFor()` - 条件等待
   - `expectReject()` - 异常断言

### 影响范围

**新增文件**：
- `tests/utils/test-database.ts` (88 行)
- `tests/utils/mock-factories.ts` (76 行)
- `tests/utils/test-helpers.ts` (55 行)
- `tests/integration/database.integration.test.ts` (148 行，14 个用例)
- `tests/e2e/system-lifecycle.e2e.test.ts` (145 行，9 个用例)

**测试覆盖**：
- ✅ messages 表 CRUD 与事务完整性
- ✅ registered_groups 唯一约束验证
- ✅ scheduled_tasks 排序查询
- ✅ 数据库表完整性与 pragma 验证
- ✅ 消息接收→处理完整流程
- ✅ 多通道并发写入
- ✅ 群组注册→停用生命周期
- ✅ 路由器状态读写

**无破坏性变更** - 仅新增测试代码，不影响现有功能

### 风险评估

**风险**: 低

**缓解措施**: 
- 测试代码与生产代码隔离
- 使用独立的测试数据库
- 所有测试在事务中运行，自动回滚

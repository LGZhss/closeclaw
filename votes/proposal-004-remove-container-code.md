## 📋 提案说明

### 背景

当前项目中包含 Docker 容器化部署相关的代码和配置（`container/` 目录及相关模块），但这些功能存在以下问题：

1. **依赖 Anthropic API** - 容器内运行 Claude Code CLI 需要 Anthropic API Key，而项目没有此密钥
2. **使用率极低** - 实际使用中容器功能的使用概率 < 0.5%
3. **增加复杂度** - 需要安装和维护 Docker，增加了项目的复杂度和学习成本
4. **个人使用场景** - 当前为个人项目，不需要容器隔离和多用户环境

### 目标

移除所有容器化部署相关的代码和配置，简化项目结构，降低维护成本。

### 实现方案

**删除的文件和目录**：
1. `container/` - 整个目录（Dockerfile、build.sh、agent-runner/）
2. `src/container-runner.ts` - 容器运行器模块

**修改的文件**：
1. `src/index.ts` - 移除容器调用逻辑
2. `src/config.ts` - 移除容器相关配置常量
3. `setup/index.ts` - 移除容器构建步骤
4. `docs/reference/FILE_STRUCTURE.md` - 更新文件结构说明
5. `groups/global/CONTEXT.md` - 更新架构文档

**保留的功能**：
- 直接在本地执行任务（不通过容器）
- 其他所有核心功能保持不变

### 影响范围

**删除的代码**：
- 约 200-300 行容器相关代码
- 占项目总代码量的 ~0.5%-1%

**影响的模块**：
- ✅ `src/index.ts` - 简化 Agent 调用逻辑
- ✅ `src/config.ts` - 移除容器配置项
- ✅ `setup/` - 简化安装流程
- ❌ **不影响** - 消息处理、投票系统、任务调度等核心功能

**破坏性变更**：
- ⚠️ 如果未来需要容器功能，需要重新添加这些代码
- ⚠️ 但可以通过其他方式实现（如直接调用本地 CLI）

### 风险评估

**风险**: 
- 低 - 这些功能当前就无法使用（缺少 API Key）
- 低 - 不影响现有可用功能

**缓解措施**: 
- 在文档中明确说明已移除容器功能
- 如有需要，可以通过 Git 历史恢复这些代码
- 未来可通过插件机制支持容器功能

## 💬 讨论区

**发起者说明**: 

这个提案的目的是简化项目，移除那些：
1. 无法使用的功能（需要 Anthropic API）
2. 增加复杂度但不产生价值
3. 99% 的场景下不会用到

如果你同意这个修改，请在投票表中填写你的态度。如果有疑问或建议，欢迎在讨论区提出！

### Antigravity 的审计意见

✅ **赞同理由**：

1. **架构减负**：Docker 依赖对于当前的个人使用场景（特别是无法访问原 Anthropic API 时）属于过度设计。
2. **维护性优化**：移除未被使用的代码（Dead Code）能显著减少 CI 负担和 Agent 的认知负荷（Token 消耗）。
3. **可逆性**：此项重构通过 Git 历史完全可溯，未来若需重新引入容器化，可在新架构下以插件形式实现。
